# SDD（规范驱动开发）经验分享汇总

> 数据来源：网络公开文章整理
> 整理时间：2026-05-13

---

## 一、核心理念

SDD（Specification-Driven Development，规范驱动开发）的核心思想是：**先写规范，再写代码。规范是合同，代码是履约。** 规范成为唯一可信源（Single Source of Truth），AI 成为高效执行者。

## 二、大厂/知名团队实践案例

### 1. 阿里巴巴 — 多 AI 协同 + SDD 全流程交付

- **来源**：阿里技术官方账号（2026-01）
- **实践**：在跨境保险复杂业务场景中，构建了由 Claude Code、CodeX 与 Gemini 协同驱动的 AI Coding 工作流
- **核心方法**：多模型协同互补 + 规范约束体系，从零交付完整产品
- **痛点解决**：避免单一模型能力不足、减少 Prompt 调试时间

### 2. AWS — Kiro（SDD-Native IDE）

- 明确把"规范"作为从原型到生产的桥梁
- 工作流：Brainstorm → Map → Act → Develop，每行代码可回溯到 Story，Story 可回溯到 PRD

### 3. GitHub — Spec-Kit

- 核心理念：先写规格，再写代码
- 特点：通过"宪法"文件约束技术决策，防止架构腐化
- 适合：架构一致性要求高、质量门控需求强、跨职能团队协作

## 三、标准工作流（四阶段 SOP）

| 阶段 | 目标 | 人类职责 | AI 职责 |
|------|------|----------|---------|
| **Specify（编写规范）** | 确立业务目标与用户价值 | 提出产品愿景与业务需求 | 转化为结构化规范文档 |
| **Plan（方案设计）** | 确定技术实现路径 | 提供技术栈、架构约束 | 生成技术方案与备选方案 |
| **Tasks（任务拆解）** | 解构为可独立执行单元 | 审核合理性 | 拆解为原子级任务清单 |
| **Implement（AI 实现）** | 生成符合规范的代码 | 审查质量与规范符合度 | 自动生成代码和测试 |

## 四、高质量规范文档 5 大要素

1. **目标与价值**：明确业务问题和收益
2. **上下文与约束**：技术栈、性能要求、安全规范
3. **功能需求**：业务流程、字段规则、异常处理
4. **非功能需求**：响应时间、并发能力、权限控制
5. **测试标准**：功能用例、边界用例、验收规则

## 五、三种实施层级（渐进式）

| 层级 | 定义 | 类比 |
|------|------|------|
| **Spec-first（规范优先）** | 先写规范再开发，规范短期有效 | 先写菜谱再做菜 |
| **Spec-anchored（规范锚定）** | 规范长期存在于仓库，随功能同步更新 | 施工图纸持续修订 |
| **Spec-as-source（规范即源码）** | 人类只编辑规范，代码完全由 AI 生成 | 规范就是源代码 |

## 六、团队实践关键经验

- **人类决策点（Human Checkpoint）** 是生命线：规范、计划、任务都需要人工审阅确认后再让 AI 执行
- **规范即"宪法"**：通过 `.cursorrules` 或全局 Prompt 将团队规范转化为机器强制执行的底线
- **变更即提案**：任何修改都创建变更提案文件夹，而非直接在代码上改
- **对抗验收**：多模型交叉验证，避免单一模型偏差
- **原子化提交**：每个任务独立提交，保证可追溯性

## 七、主流工具对比

| 工具 | 特点 | 适合场景 |
|------|------|----------|
| **OpenSpec** | 灵活轻量，Markdown + 斜杠命令 | 存量项目增量引入，上手成本低 |
| **Spec-Kit** | 文档规范完整，有宪法门禁 | 新项目、架构一致性要求高 |
| **Superpowers** | 多 Agent 虚拟团队协作 | 复杂项目的多模型协同 |
